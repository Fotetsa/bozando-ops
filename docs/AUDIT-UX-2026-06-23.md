# Audit DevOps — Interface web bozando-ops

Date : 2026-06-23
Périmètre : `packages/web` (SPA Vite + React Flow + Medusa UI), de bout en bout.
Posture : production (cf. règle « prod, pas test »). Référentiels UX : Azure Portal, Railway, GNS3.

Légende sévérité :
- P0 — bloquant / l'outil ne peut pas remplir sa mission ou casse en prod
- P1 — gouffre fonctionnel ou d'accessibilité majeur
- P2 — écart UX / cycle de vie incomplet
- P3 — finition

---

## 1. Onboarding & cycle de vie de l'installation (P0)

| # | Constat | Sévérité |
|---|---|---|
| 1.1 | **Aucune page de bootstrap.** La route `POST /api/auth/bootstrap` (création du 1er owner) existe mais n'est appelée nulle part dans `web/`. Un déploiement neuf affiche `LoginPage` sans compte possible → impasse, l'admin doit curler à la main. | P0 |
| 1.2 | `LoginPage` ne distingue pas « identifiants faux » de « aucun compte n'existe encore ». Pas de redirection automatique vers le bootstrap quand 0 user. | P0 |
| 1.3 | Aucun écran « état du système » au login (Swarm actif ? API jointe au socket ? Caddy joignable ?). L'opérateur découvre les pannes d'infra une fois dans le canvas. | P1 |

## 2. RBAC invisible côté UI (P0/P1)

| # | Constat | Sévérité |
|---|---|---|
| 2.1 | Le backend applique `requireRole(owner/operator/viewer)` mais l'UI **n'en tient aucun compte** : la sidebar montre tous les liens à tous, les boutons mutants (Déployer, Provisionner, Prune, Secrets) s'affichent pour un `viewer` → clic → 403 opaque toast « Erreur ». But déclaré du projet = **déléguer à un employé** : c'est précisément le rôle `operator/viewer` qui doit avoir une UI cohérente. | P0 |
| 2.2 | **Aucune gestion d'utilisateurs.** Pas de route `/api/users` ni d'écran : impossible de créer l'employé délégué depuis l'UI alors que c'est la raison d'être de l'outil. La table `User` (role) existe en base mais est inexploitable. | P0 |
| 2.3 | `me.role` est chargé dans Settings mais jamais propagé en contexte global pour gater l'UI. | P1 |

## 3. Audit / traçabilité (P1)

| # | Constat | Sévérité |
|---|---|---|
| 3.1 | La table `AuditLog` est **écrite** (subscriber `on-deploy-finished`) mais **jamais lue** : pas de route GET, pas d'écran. Or pour « déléguer des ops risquées à un employé » la traçabilité « qui a déployé/détruit quoi quand » est non-négociable (équivalent Activity Log Azure). | P1 |

## 4. Canvas — gouffres logiques & cycle de vie (P0/P1)

| # | Constat | Sévérité |
|---|---|---|
| 4.1 | **Pas de feedback désiré-vs-réel sur le canvas.** Un nœud ajouté/édité après un deploy ne montre nulle part qu'il est « non déployé / en attente ». La pastille d'état (running/exited/missing) ne dit pas « le désiré diverge du réel » → l'opérateur ne sait pas qu'il doit redéployer. GNS3/Railway montrent toujours l'écart. | P0 |
| 4.2 | **Déployer/Détruire sans garde-fou ni résumé.** Pas de confirmation, pas de « plan » (la route `GET /api/projects/:id/plan` existe et n'est PAS utilisée par l'UI). On clique Détruire et tout part. Azure/Railway montrent toujours un diff avant apply. | P0 |
| 4.3 | **Pas de drag pour créer un nœud au clavier / pas d'alternative non-drag.** La palette est exclusivement drag-and-drop HTML5 → inaccessible clavier, inutilisable tactile fiable, et zéro fallback (clic pour ajouter au centre). | P1 (a11y) |
| 4.4 | Le canvas n'expose **ni undo/redo, ni copier-coller, ni suppression au clavier (Del)** d'un nœud/edge sélectionné. React Flow le permet, ce n'est pas câblé. | P2 |
| 4.5 | **Pas de validation de cohérence avant deploy** : un conteneur sans réseau, une passerelle sans cible, un volume orphelin ne sont pas signalés visuellement. L'erreur tombe au deploy. | P1 |
| 4.6 | L'erreur de déploiement est un toast unique tronqué (`r.log.length` opérations) — **pas de panneau d'activité/log de déploiement** consultable, pas d'historique des runs. Railway a un onglet Deployments. | P1 |
| 4.7 | `onConnectEnd` toast « impossible » uniquement si `fromNode && toNode` : un drag lâché dans le vide ne dit rien (silencieux). | P3 |
| 4.8 | Pas de mini-instructions / état vide guidé sur un projet neuf (canvas vide = écran noir déroutant). Railway/GNS3 ont un empty-state pédagogique. | P2 |

## 5. Accessibilité (P1) — « défi réel » signalé par l'utilisateur

| # | Constat | Sévérité |
|---|---|---|
| 5.1 | **Aucun attribut ARIA / rôle** sur les conteneurs interactifs custom (palette draggable = `div` sans `role`, sans `tabindex`, sans `aria-grabbed`). | P1 |
| 5.2 | Le burger mobile et les overlays (Inspector, Palette) ne piègent pas le focus, pas d'`aria-label`, pas de fermeture `Esc`. | P1 |
| 5.3 | États communiqués **uniquement par la couleur** (pastille running/exited/missing) sans texte/forme alternatif → invisible pour daltoniens. WCAG 1.4.1. | P1 |
| 5.4 | `IconButton` à action seule (burger, fermer, action menu) sans `aria-label` → lecteur d'écran annonce « button ». | P1 |
| 5.5 | Le `<pre>` de logs/provisioning n'a pas `aria-live` → contenu streamé jamais annoncé. | P2 |
| 5.6 | Contraste des `txt-compact-xsmall text-ui-fg-muted` sur fond subtil à vérifier (probable < 4.5:1). | P2 |
| 5.7 | Pas de skip-link, pas de gestion `:focus-visible` cohérente, ordre de tabulation non maîtrisé sur le canvas. | P2 |

## 6. Robustesse front / contrats (P1/P2)

| # | Constat | Sévérité |
|---|---|---|
| 6.1 | `main.tsx` instancie `new QueryClient()` **sans defaults** : pas de `retry` borné, pas de `staleTime`, les erreurs réseau retentent en boucle (atténué seulement pour les 401). | P2 |
| 6.2 | Pas d'**Error Boundary** : une exception de rendu (ex. donnée inattendue) écran blanc total. | P1 |
| 6.3 | `useOpsSocket` : `useEffect` dépend de `onNodeState/onNodeReplicas` (callbacks recréés) → reconnexions socket potentielles ; pas d'indicateur « temps réel connecté/perdu ». L'opérateur ne sait pas si le canvas est « live » ou figé. | P1 |
| 6.4 | Aucun état de **chargement/erreur** sur `getProject` dans le canvas (si l'appel échoue, titre « … » indéfiniment). | P2 |
| 6.5 | `provisionLog` rendu avec `key={i}` et `.map(...).join("")` superflu ; pas d'auto-scroll. | P3 |
| 6.6 | Pas de `<title>` par page ni de meta — onglets indistinguables. | P3 |

## 8. Cohérence visuelle / design system (P2/P3)

| # | Constat | Sévérité |
|---|---|---|
| 8.1 | Pages de liste hétérogènes : `ListContainer` existe mais **Projects/Servers/Secrets/Integrations ne l'utilisent pas** (chacune réinvente sa liste). Incohérence d'espacement/typo. | P2 |
| 8.2 | Empty-states inégaux (« Aucun projet. Crées-en un. » vs rien sur le canvas). | P3 |

## Synthèse priorisée (ce qui DOIT être fait)

**P0 — l'outil ne remplit pas sa mission sans ça :**
1. Bootstrap onboarding (1.1, 1.2)
2. RBAC effectif dans l'UI + gestion d'utilisateurs (2.1, 2.2, 2.3)
3. Diff désiré-vs-réel sur le canvas + plan avant deploy + garde-fou destroy (4.1, 4.2)

**P1 — gouffres majeurs / accessibilité :**
4. Audit log lisible (3.1)
5. Accessibilité de base : ARIA, focus, Esc, alternatives couleur, palette accessible (5.x, 4.3)
6. Error boundary + indicateur temps-réel + santé infra au login (6.2, 6.3, 1.3)
7. Validation de cohérence du graphe + panneau d'activité de déploiement (4.5, 4.6)

**P2/P3 — cycle de vie complet & finition :**
8. QueryClient defaults, états chargement/erreur (6.1, 6.4)
9. Drill-down métriques service (7.1)
10. Homogénéisation listes via ListContainer + empty-states + densité OpsNode (8.1, 8.2, 8.3)
11. Canvas : Del/undo/copier, empty-state guidé (4.4, 4.8)
